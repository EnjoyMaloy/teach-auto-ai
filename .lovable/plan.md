## Команды (Teams) с общим доступом к курсам и инструкциям

### Концепция

Пользователь может создавать **команды** и приглашать в них других зарегистрированных пользователей по email. Курсы и инструкции, созданные внутри команды, принадлежат команде, а не одному человеку — все участники видят их в общем пространстве. Один пользователь может состоять в нескольких командах + у него остаётся личное пространство.

### Роли в команде

- **Member** — может всё, кроме удаления контента и управления составом команды (создавать, редактировать, публиковать курсы и инструкции).
- **Admin** — то же, что Member, плюс может удалять курсы/инструкции, добавлять и удалять участников, менять роли, удалять команду. Создатель команды автоматически становится Admin.

### Модель данных (новые таблицы)

1. **`teams`** — `name`, `description`, `avatar_url`, `created_by`.
2. **`team_members`** — `team_id`, `user_id`, `role` (`admin` | `member`), уникальная пара (team_id, user_id).
3. **Расширение `courses`** — добавить nullable `team_id`. Если поле заполнено, курс принадлежит команде; иначе — личный курс автора.
4. **Расширение `articles`** — добавить nullable `team_id` с той же логикой.

Существующая колонка `author_id` / `user_id` сохраняется как «кто создал», но права доступа определяются через `team_id` + `team_members`.

### Безопасность (RLS)

Создаём security definer функции:
- `is_team_member(_team_id, _user_id)` — проверка участия.
- `is_team_admin(_team_id, _user_id)` — проверка роли admin.

Обновляем RLS на `courses`, `articles`, `lessons`, `slides`, `lesson_translations`, `slide_translations`, `published_*`, `course_languages`, `course_ai_messages`:

- **SELECT**: автор ИЛИ участник команды (`team_id` курса).
- **INSERT/UPDATE**: автор ИЛИ участник команды (member или admin).
- **DELETE**: автор ИЛИ admin команды.

`team_members` RLS:
- SELECT — все участники своей команды видят список.
- INSERT/UPDATE/DELETE — только admin команды.
- Удаление себя из команды (`leave team`) разрешено любому участнику.

`teams` RLS:
- SELECT — участники команды.
- UPDATE/DELETE — только admin.
- INSERT — любой авторизованный пользователь (создаёт команду, становится admin через триггер).

Триггер на `teams` после INSERT — автоматически добавляет создателя в `team_members` с ролью `admin`.

### UI

1. **Новая страница `/teams`** в основной навигации (sidebar):
   - Список команд пользователя (карточки).
   - Кнопка «Создать команду» → диалог с name + description.
   - Клик по команде → детальная страница `/teams/:teamId`.

2. **Страница команды `/teams/:teamId`**:
   - Шапка: название, описание, аватар команды (для admin — редактируемые).
   - Таб «Участники»: список с email, ролью и аватаром. Для admin — кнопка «Добавить участника» (ввод email уже зарегистрированного пользователя; если такого нет — toast «Пользователь должен сначала зарегистрироваться»), смена роли, удаление.
   - Таб «Курсы»: курсы команды (используем существующую `CourseCard`).
   - Таб «Инструкции»: статьи команды.

3. **Селектор пространства (workspace switcher)**:
   - Выпадающий селектор в шапке/sidebar: «Личное пространство» + список команд.
   - Выбранное пространство хранится в localStorage и определяет:
     - какие курсы/инструкции показываются на `/workshop`, `/articles`, `/favorites`;
     - в какое пространство попадают новые курсы (`team_id` = выбранная команда или `null`).

4. **Индикатор владельца** на карточках курса/инструкции — мелкий бейдж с названием команды или иконкой «личное».

### Технические детали

- Миграция: новые таблицы + колонки `team_id` + RLS-функции + переписанные политики.
- Хук `useTeams()` (список команд пользователя) и `useCurrentWorkspace()` (контекст активного пространства).
- При создании курса/инструкции в `Editor` / `Articles` подставлять `team_id` из активного workspace.
- Поиск пользователя по email через `profiles` (email уже хранится). Если профиля нет — отказ.
- Запрет на удаление последнего admin команды (чтобы команда не осталась без админа).

### Что НЕ делаем в этой итерации

- Email-уведомления о приглашении (добавляем только если пользователь уже есть в системе).
- Шаринг отдельных курсов «точечно» вне команды — только через команду.
- История активности команды.
